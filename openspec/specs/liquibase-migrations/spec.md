# Liquibase Migrations

## Purpose
Replace bash-based database migration scripts with Liquibase for applying EF Core-generated SQL migrations in sandbox and production environments, while preserving the EF Core migration authoring workflow.

## Requirements

### Requirement: Liquibase changelog per database
The system SHALL maintain one Liquibase XML changelog per database (auth-db, permission-db, expense-db) under `docker/liquibase/{db}/changelog.xml`, with per-migration changeset files in a `changesets/` subdirectory.

#### Scenario: Changelog structure exists for all databases
- **WHEN** the repository is cloned
- **THEN** `docker/liquibase/auth-db/changelog.xml`, `docker/liquibase/permission-db/changelog.xml`, and `docker/liquibase/expense-db/changelog.xml` SHALL exist

### Requirement: EF Core SQL referenced as Liquibase changesets
Each Liquibase changeset SHALL reference the corresponding EF Core-generated SQL file via `<sqlFile>`. EF Core migrations remain the source of truth for schema authoring.

#### Scenario: Changeset references EF-generated SQL
- **WHEN** a Liquibase changeset is created for an EF Core migration
- **THEN** the changeset SHALL use `<sqlFile>` to point to the SQL file in `docker/migrations/{db}/` generated by `dotnet ef migrations script`

#### Scenario: New migration authoring workflow
- **WHEN** a developer runs `just add-migration-auth "AddFeature"`
- **THEN** an EF Core C# migration class SHALL be created (unchanged from current behavior)
- **AND** a corresponding Liquibase changeset XML SHALL be created in `docker/liquibase/auth-db/changesets/` referencing the SQL output

### Requirement: Sandbox deployment via Docker Compose
The system SHALL use a Liquibase Docker container in `docker-compose.sandbox.yml` to apply migrations before API services start.

#### Scenario: Migrations run on sandbox startup
- **WHEN** `docker compose -f docker-compose.sandbox.yml up` is executed
- **THEN** the Liquibase container SHALL apply all pending changesets to all 3 databases before any API container starts

#### Scenario: API services depend on migration completion
- **WHEN** the Liquibase container has not yet completed
- **THEN** API containers SHALL not start (enforced via `depends_on: condition: service_completed_successfully`)

### Requirement: Justfile Liquibase recipes
The system SHALL provide justfile recipes that wrap Liquibase CLI commands for migrate, rollback, and status operations.

#### Scenario: Apply migrations for a single database
- **WHEN** `just migrate-auth` is executed
- **THEN** all pending changesets for auth-db SHALL be applied via `liquibase update`

#### Scenario: Apply migrations for all databases
- **WHEN** `just migrate-all` is executed
- **THEN** all pending changesets for auth-db, permission-db, and expense-db SHALL be applied

#### Scenario: Rollback a specific number of changesets
- **WHEN** `just rollback-auth` is executed
- **THEN** the last applied changeset for auth-db SHALL be rolled back via `liquibase rollback-count 1`

#### Scenario: List migration status
- **WHEN** `just list-migrations-auth` is executed
- **THEN** the system SHALL display the Liquibase status for auth-db showing applied and pending changesets

### Requirement: EF Core authoring tooling preserved
The system SHALL keep `dotnet-ef` in the development container and preserve all EF Core migration C# files. The `dotnet ef migrations add` workflow is unchanged.

#### Scenario: dotnet-ef available in dev container
- **WHEN** the dev container is built from `Dockerfile.dev`
- **THEN** `dotnet ef` command SHALL be available

#### Scenario: EF Core migration files retained
- **WHEN** the repository is at the new state
- **THEN** all `src/**/Migrations/*.cs` files SHALL still exist

### Requirement: Integration tests unchanged
The system SHALL NOT modify any integration test code. Tests continue using `MigrateAsync()` / `EnsureCreatedAsync()` via EF Core directly.

#### Scenario: Test fixtures use EF Core directly
- **WHEN** integration tests run
- **THEN** `ExpenseApiFixture`, `PermissionApiFixture`, `AuthApiFixture`, and `RoleSyncFixture` SHALL continue calling `Database.MigrateAsync()` or `Database.EnsureCreatedAsync()` without changes

### Requirement: Bash migration scripts removed
The system SHALL remove `run-migrations.sh`, `run-rollback.sh`, and `000-init.sh` from `docker/migrations/`. The `docker/migrations/` directory structure may be retained for SQL file output only.

#### Scenario: Old scripts removed
- **WHEN** the repository is at the new state
- **THEN** `docker/migrations/run-migrations.sh`, `docker/migrations/run-rollback.sh`, and `docker/migrations/000-init.sh` SHALL not exist

### Requirement: DATABASECHANGELOG tracking in sandbox
The system SHALL use Liquibase's `DATABASECHANGELOG` and `DATABASECHANGELOGLOCK` tables for tracking applied changesets in sandbox/production databases.

#### Scenario: Liquibase tracking tables created
- **WHEN** Liquibase first connects to a sandbox database
- **THEN** `DATABASECHANGELOG` and `DATABASECHANGELOGLOCK` tables SHALL be created in the `public` schema

#### Scenario: Baseline for existing databases
- **WHEN** Liquibase connects to a database that already has schema from prior EF Core migrations
- **THEN** a baseline changeset SHALL mark all existing migrations as applied to prevent re-execution
